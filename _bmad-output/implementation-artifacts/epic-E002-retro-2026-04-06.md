# Epic E002 Retrospective — Application CRUD

**Epic:** `E002`  
**Date:** April 6, 2026  
**Status:** done

## Scope Shipped

- OpenAPI-backed application CRUD contract
- Create/list/get/update/delete flows in `job-service`
- Search, filtering, sorting, and pagination behavior
- Ownership enforcement on all application routes
- Integration coverage for happy + denial paths

## Wins

- Contract-first approach kept CRUD shape stable while implementation iterated
- Ownership constraints were simple to reason about because `user_id` was present on every application record
- Pagination/filtering landed without requiring additional gateway complexity

## Pain Points

- In-memory store implementation is fine for tests but hides production persistence concerns
- Search/sort semantics were implemented quickly but need query/index review before scale work
- CRUD and parse-url concerns share the same service boundary, which may grow noisy over time

## Decisions That Helped

- Explicit `authRequired` + downstream ownership checks
- Keeping gateway orchestration thin and letting `job-service` own record filtering
- Normalized `ErrorEnvelope` responses for not-found/forbidden cases

## Action Items

1. Add composite DB indexes for `user_id`, `status`, and `created_at` in Wave 2 performance work
2. Define persistence-backed repository layer before high-scale testing
3. Add contract examples for list filtering and sorting edge cases
