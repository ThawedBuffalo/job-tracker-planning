# Epic E004 Retrospective — Pipeline Transitions & Timeline

**Epic:** `E004`  
**Date:** April 6, 2026  
**Status:** done

## Scope Shipped

- Transition validation rules in `pipeline-service`
- Append-only timeline events
- Idempotent transition handling with outbox compatibility
- Timeline query with deterministic `next_step_prompt`
- Stalled reminder scan support foundation

## Wins

- Transition policy table kept business rules readable and testable
- Append-only timeline model made later reminder/event work straightforward
- Idempotency support reduced duplicate-event risk early

## Pain Points

- In-memory event/outbox storage is great for tests but not sufficient for durability
- Terminal-state naming in stories vs code (`closed` vs `accepted`) created a terminology mismatch
- Timeline and stalled detection now share store logic that will need persistence cleanup later

## Decisions That Helped

- Treat timeline as immutable event history
- Generate deterministic guidance text from status enum rather than ad hoc copy
- Keep protected ownership checks inside the service even after gateway enforcement

## Action Items

1. Persist outbox/timeline events in database-backed tables
2. Normalize status vocabulary across stories, code, and contracts
3. Add scheduler abstraction before productionizing stalled scans
