# Epic E006 Retrospective — Notifications & Reminder Loop

**Epic:** `E006`  
**Date:** April 6, 2026  
**Status:** done

## Scope Shipped

- Notification contracts/events defined
- Dispatch endpoint and consumer flow in `notification-service`
- SMTP adapter and templated emails
- Retry/backoff with delivery attempt logging
- Stalled application reminder consumption with cadence controls

## Wins

- Retry executor abstraction kept sync dispatch and consumer sends aligned
- Correlation-id based idempotency avoided duplicate notification records
- Reminder cadence and opt-out handling were added without rewriting the core send flow

## Pain Points

- Delivery storage is still in-memory, so operational realism is limited
- Notification dispatch remains synchronous from API perspective even though retry semantics exist
- Preference model is intentionally small and will need richer per-template controls later

## Decisions That Helped

- Separate mailer adapter from consumer logic
- Persist attempt history per notification ID
- Use common retry executor for all notification paths

## Action Items

1. Move notifications/attempt logs to durable persistence
2. Introduce queue-backed async delivery for throughput and reliability
3. Expand preference schema beyond channel-level toggles

## Wave 2 Carry-Forward

- Directly informs `E011` async delivery work
- Supports observability requirements in `E014`
- Supports compliance/audit enhancements in `E013`
